> 本文是 Go 源码中文档（ `cmd/compile/README.md` ）的翻译。

`cmd/compile` 中包含了组成 Go 编译器的主要包。编译器可以从逻辑上划分为四个阶段，接下来会分别进行介绍。

众所周知，编译器可以分为前端和后端。粗略地讲，前两个阶段可以算作前端，后两个阶段可以算作后端。也有中端这种说法，通常指的是第二阶段。

注意 `go/*` 系列的包例如 `go/parser` 、 `go/types` ，它们与编译器没有关系。因为编译器一开始时是用 C 编写的，为了开发处理 Go 代码的工具例如 `gofmt` 和 `vet` ，这才有了 `go/*` 。

还有一点要注意，在本文以及后续系列文档中， `gc` 指的是 Go 编译器（ Go Compiler ），而 GC 指的是垃圾回收（ Garbage Collection ）。

## 1. 解析

`cmd/compile/internal/syntax` ：词法分析器，文法解析器，语法树。

在编译的第一阶段，源代码被 token 化（ tokenize ，也称为词法分析）、解析（文法分析），最后每一个源文件都变成一个语法树。

每个语法树都是对应源文件的精确表示，树中每个节点都对应源代码中的元素，例如表达式、声明和陈述。语法树中还包含位置信息，以便于错误报告和调试。

## 2. 类型检查和 AST 转换

`cmd/compile/internal/gc` ：创建编译器 AST ，类型检查， AST 转换。

`gc` 包中包含一个从 C 代码时代遗留下来的 AST 定义。所有其他的代码都是基于它编写的，所以第一件事就是把 `syntax` 的语法树转换成 `gc` 包中的语法树。这多余的一步在未来的重构应该会被移除。

然后会进行类型检查。第一步是名称解析和类型推断，判断哪个对象属于哪个标识符，每个表达式的类型是什么。类型检查包含了一些特定的额外检查，比如“定义但未使用”、函数是否可终止等。

在 AST 检查完成后还会进行一些转换。某些节点会根据类型信息进行重构，比如字符串加法会与算术加法区别开来。另外，死代码消除、函数调用内联、逃逸分析等也会在这里完成。

## 3. 通用 SSA

`cmd/compile/internal/gc` 转换为 SSA ， `cmd/compile/internal/ssa` SSA 遍历和规则。

在本阶段中， AST 被转换为静态单赋值（ SSA ， Static Single Assignment ）形式。 SSA 是一种低级中间表示，其性质使得它能够更好地进行优化，并最终生成机器码。

在这个阶段，函数内联被应用。这是一类特殊函数，它们对应的 SSA 已经被高度优化过，会被直接匹配并替换。

某些节点会被降级成更简单的组分，使接下来的工作更加方便。比如，内建的 `copy` 操作会被替换成内存移动， `range` 循环会被重写成 `for` 循环。不过，某些转换发生在转换为 SSA 之前，这是历史遗留问题，长期计划中它们都应该被放在这里。

然后，一系列机器无关的遍历和规则将会被应用。这些操作不关心单一的计算机架构，因此会在所有的 `GOARCH` 下执行。

在遍历过程中将会进行一些优化，比如死代码消除、不必要的 `nil` 检查消除、无用分支的消除。而规则更专注于表达式，例如使用常量替换表达式，优化乘法和浮点操作等。

## 4. 生成机器码

`cmd/compile/internal/ssa` SSA 降级和特定架构下的遍历， `cmd/internal/obj` 机器码生成。

机器相关的阶段从降级遍历开始，把通用形式重写为特定机器下的变体。比如，在 `amd64` 架构下可以使用内存操作数，许多存取操作可以被其他操作吸收。

降级遍历将会执行所有的机器特定重写规则，因此有许多优化规则也包含在内。

SSA 降级为特定架构下的代码后，将会执行最终的代码优化遍历，包括死代码消除、把值移动到距离使用更近的地方、移除从未读取的局部变量、寄存器分配。

这一步中其他的重要工作包括：栈帧布局，将栈偏移量赋值给局部变量；指针生命周期分析，计算哪些栈中的指针在 GC 安全点是活跃的。

在 SSA 生成阶段的最后， Go 函数被转换成一系列 `obj.Prog` 指令。这些指令将会被送给汇编器（ `cmd/internal/obj` ），由汇编器将其转换为机器码，并形成最终的目标文件。目标文件还会包含反射数据、导出数据和调试信息。
