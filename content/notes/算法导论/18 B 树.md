**B 树**（ B-tree ）是一种平衡搜索树，设计目的是在硬盘这种次级存储设备上获得更好的性能。 B 树和红黑树很像，但是 B 树的设计原则倾向于最小化硬盘的 I/O 次数。许多数据库都使用 B 树或者 B 树的变体来存储信息。

B 树与红黑树不同的地方在于 B 树节点可以有许多孩子，从几个到几千个不等，取决于硬盘的某些特性。 B 树与红黑树类似的地方在于 $n$ 个节点的 B 树的高度是 $O(\lg n)$ 。这使得 B 树的很多操作可以在 $O(\lg n)$ 的运行时间内完成。

![一棵 B 树](/notes/算法导论/images/18-00.png#center)

上图中展示了一棵 B 树。假如某个 B 树节点 $x$ 中有 $x.n$ 个键，那么 $x$ 就会有 $x.n+1$ 个孩子。 B 树中的叶节点和非叶节点的结构不同，具体差别后面会详细介绍。

目前大型数据库的主要存储载体还是机械硬盘。顾名思义，机械硬盘中仍然含有大量机械结构，相比于纯电路结构，这些机械结构慢得多。机械硬盘内部的结构类似于黑胶唱片的播放器，也就是老式留声机的结构。记录数据的磁盘快速转动，磁头则可以在盘面上移动，寻找需要读写的数据的位置。当需要读取数据时，首先磁头需要移动到对应的位置，然后还需要等待记录数据的区域旋转到磁头的位置。即使磁盘以每分钟五千转以上的速度高速旋转，但是这些机械活动相对于高频的 CPU 来说还是太慢了。为了提高效率，磁盘每次会读取一**页**（ page ）的数据。每一页都是磁盘上连续的一小段，可能含有几百乃至上千字节的数据。

## 17.1 B 树的定义

通常， B 树中都不会保存实际的数据，而只会保存一个索引和一个指向实际数据的指针。在本章的讨论中，我们只关注索引，略去指针的部分。

B 树 $T$ 是以 $T.root$ 为根节点的树，具有以下性质：

1. 任意节点 $x$ 都有以下属性：
   1. $x.n$ 代表本节点中保存的键的数量。
   2. 以非递减顺序保存的 $x.n$ 个键， $x.key_1 \leqslant x.key_2 \leqslant \dots \leqslant x.key_{x.n}$ 。
   3. $x.leaf$ 中保存了本节点是否是一个叶节点。
2. 每个内部节点 $x$ 含有 $x.n + 1$ 个指向它的孩子的指针 $x.c_1, x.c_2, \dots, x.c_{x.n+1}$ 。叶节点没有孩子。
3. 键 $x.key_i$ 把键值范围分割成了数个区域，假设 $k_i$ 是子树 $x.c_i$ 中储存的任意键，那么有 $k_1 \leqslant x.key_1 \leqslant k_2 \leqslant x.key_2 \leqslant \dots \leqslant x.key_{x.n} \leqslant k_{x.n+1}$ 。
4. 所有叶节点都位于同一深度，等于树的高度 $h$ 。
5. 每个节点中含有的键的数量有上界和下界。设某个整数 $t \geqslant 2$ 是这个 B 树的**最小度**（ minimum degree ）：
   1. 除了根节点之外的任何节点最少要有 $t-1$ 个键，除了根节点的任何非叶节点至少要有 $t$ 个孩子。如果树非空，那么根节点至少要有一个键。
   2. 任何节点最多有 $2t - 1$ 个键，内部节点最多有 $2t$ 个孩子。

最简单的 B 树中 $t = 2$ ，每个内部节点可以有 2 、 3 或者 4 个孩子，因此也被称为 2-3-4 树。在实际使用的 B 树中 $t$ 要大得多。

### B 树的高度

大多数 B 树的操作的磁盘访问次数与 B 树的高度成正比，我们分析一下最坏情况下 B 树的高度。

### 定理 18.1

如果 $n \geqslant 1$ ，那么对于任何含有 $n$ 个键的、高度为 $h$ 的、最小度 $t \geqslant 2$ 的 B 树来说：

$$
h \leqslant \log_t \frac{n+1}{2}
$$

假设 B 树的根节点含有至少一个键，根节点之外的节点含有至少 $t-1$ 个键。那么在 $T$ 中，深度为 1 的节点至少有 $2$ 个，深度为 2 的节点至少有 $2t$ 个，深度为 3 的节点至少有 $2t^2$ 个，以此类推，深度为 $h$ 的节点至少有 $2t^{h-1}$ 个。综上所述，可以写出不等式：

$$
\begin{aligned}
n & \geqslant 1 + (t - 1) \sum_{i=1}^{h} 2 t^{i-1} \\\\
& = 1 + 2 (t - 1) \frac{(t^h - 1)}{(t - 1)} \\\\
& = 2 t^h - 1
\end{aligned}
$$

由于 $t$ 是一个常数，所以最终 B 树的高度是 $O(\lg n)$ 。

## 18.2 B 树的基本操作

本章将会介绍三种基本操作 `B-TREE-SEARCH` 、 `B-TREE-CREATE` 和 `B-TREE-INSERT` 。为了能够在算法中显式地体现出硬盘 I/O 的次数，我们进行如下规定：

- 根节点永远位于主存或者说内存之中，访问根节点时不需要进行 `DISK-READ` ；不过，如果根节点被修改，还是需要 `DISK-WRITE` 的。
- 访问其他任何节点之前都需要进行 `DISK-READ` 。

本小节中介绍的算法都是一遍（ one-pass ）算法，只会从根节点不断向下访问子节点，而不会访问父节点。

### B 树的搜索

``` plaintext
B-TREE-SEARCH(x, k)
  i = 1
  while i <= x.n and k > x.key[i]
    i = i + 1
  if i <= x.n and k == x.key[i]
    return (x, i)
  else if x.leaf
    return nil
  else
    DISK-READ(x.c[i])
    return B-TREE-SEARCH(x.c[i], k)
```

`B-TREE-SEARCH` 使用了线性搜索的算法，找到节点 $x$ 中第一个满足 $k \leqslant x.key_i$ 的键。如果这个键就是要找的目标，那么返回当前节点，否则就在对应的子节点中寻找。

根据 `B-TREE-SEARCH` 算法的流程，访问磁盘的次数等于 $O(h) = O(\log_t n)$ 。由于 $x.n < 2t$ ，所以算法的 CPU 运行时间是 $O(th) = O(t \log_t n)$ 。

### 创建新的 B 树

``` plaintext
B-TREE-CREATE(T)
  x = ALLOCATE-NODE()
  x.leaf = TRUE
  x.n = 0
  DISK-WRITE(x)
  T.root = x
```

创建新的 B 树需要 $O(1)$ 的磁盘访问和 $O(1)$ 的 CPU 运行时间。

### 向 B 树中插入

向 B 树中插入新键显然要比在 B 树中搜索复杂得多。与搜索相同，向 B 树中插入新键时，首先要做的也是搜索，找到一个可以放下新键的叶节点。如果该叶节点已经满了，那么可以将其分裂成两个节点。一个节点的分裂会导致其父节点中需要多容纳一个键，如果分裂前父节点已满，那么子节点分裂后，父节点也需要分裂，层层递归。如果最后根节点也分裂，那么就会创建新的根节点，整棵 B 树就会这样长高一层。

然而，上面的分裂方法会使得算法访问其父节点，从而违背我们的一遍算法设计原则。因此，实际的算法是这样的：在搜索的过程中，只要我们遇到了满的节点，就将其分裂为两个。如此以来，子节点的分裂就不会导致父节点的分裂了。

![节点的分裂](/notes/算法导论/images/18-01.png#center)

上图中展示了一个节点的分裂过程。具体的算法流程如下：

``` plaintext
B-TREE-SPLIT-CHILD(x, i)
  z = ALLOCATE-NODE()
  y = x.c[i]
  z.leaf = y.leaf
  z.n = t - 1
  for j = 1 to t - 1
    z.key[j] = y.key[j+t]
  if not y.leaf
    for j = 1 to t
      z.c[j] = y.c[j+1]
  y.n = t - 1
  for j = x.n + 1 downto i + 1
    x.c[j+1] = x.c[j]
  x.c[i+1] = z
  for j = x.n downto i
    x.key[j+1] = x.key[j]
  x.key[i] = y.key[t]
  x.n = x.n + 1
  DISK-WRITE(y)
  DISK-WRITE(z)
  DISK-WRITE(x)
```

首先，我们创建新的节点 `z` 。然后，把 `y` 中的后半部分数据复制到新节点中。接下来，由于需要向父节点 `x` 中插入一个节点，所以需要部分数据向后移动一位。最后，插入新节点，并将修改好的节点写入到硬盘。

观察算法可知，节点分裂需要 $O(1)$ 的磁盘访问和 $\Theta(t)$ 的 CPU 运行时间。注意 `B-TREE-SPLIT-CHILD` 假设节点已经被 `DISK-READ` 读取到了主存之中。

有了上述子程序，插入算法的流程就简洁得多：

``` plaintext
B-TREE-INSERT(T, k)
  r = T.root
  if r.n == 2*t - 1
    s = ALLOCATE-NODE()
    T.root = s
    s.leaf = FALSE
    s.n = 0
    s.c[1] = r
    B-TREE-SPLIT-CHILD(s, 1)
    B-TREE-INSERT-NONFULL(s, k)
  else
    B-TREE-INSERT-NONFULL(r, k)
```

在插入算法中，主要处理的是根节点是否分裂这种特殊情况，而具体的插入过程则体现在 `B-TREE-INSERT-NONFULL` 中：

``` plaintext
B-TREE-INSERT-NONFULL(x, k)
  i = x.n
  if x.leaf
    while i >= 1 and k < x.key[i]
      x.key[i+1] = x.key[i]
      i = i - 1
    x.key[i+1] = k
    x.n = x.n + 1
    DISK-WRITE(x)
  else
    while i >= 1 and k < x.key[i]
      i = i - 1
    i = i + 1
    DISK-READ(x.c[i])
    if x.c[i].n == 2*t - 1
      B-TREE-SPLIT-CHILD(x, i)
      if k > x.key[i]
        i = i + 1
    B-TREE-INSERT-NONFULL(x.c[i], k)
```

`B-TREE-INSERT-NONFULL` 中分两种情况，首先是当前节点是叶节点这种情况。首先需要明白一点，我们无需关心当前节点是否已满，因为算法总是提前分裂装满的节点。在本节点是叶节点的情况中，算法会将比 `k` 大的键向后移一个位置，然后将 `k` 插入到合适的位置上。在本节点是非叶节点的情况中，算法会找到对应的子节点，检查其是否需要分裂，然后递归调用。综上可知，新的键总是会被插入到叶节点。

观察算法，插入操作需要 $O(h)$ 的磁盘访问和 $\Theta(th)$ 的 CPU 运行时间。

## 18.3 从 B 树中删除

删除的操作与插入类似，但是会更复杂一些。插入操作总是会把新键放入到叶节点中，而删除操作则可能从任何节点中删除一个键。其次，删除时还需要注意节点中键的数量不能过少。

最终，删除操作的思路如下：当算法递归访问某个节点 $x$ 时，需要保证该节点中键的数量不少于 $t$ 。解决办法则是从别的节点中移动一个键过来。具体来讲，分为许多种情况，假设键 $k$ 在节点 $x$ 中：

1. 如果 $x$ 是叶节点，那么直接从 $x$ 中删除 $k$ 。
2. 如果 $x$ 是内部节点：
   1. 如果 $x$ 中 $k$ 的节点 $y$ 有至少 $t$ 个键，那么找到并删除 $k$ 的前任 $k'$ ，然后用 $k'$ 替换 $k$ 。
   2. 如果 $y$ 中的键不足 $t$ 个，那么检查 $x$ 中 $k$ 右边的节点 $z$ 。如果 $z$ 有至少 $t$ 个键，那么删除 $k$ 的后继 $k'$ ，然后用 $k'$ 替换 $k$ 。
   3. 如果 $y$ 和 $z$ 中的节点数量全都不足 $t$ 个，那么把 $k$ 和 $z$ 中的全部键都合并到 $y$ 中，然后递归地在 $y$ 中删除 $k$ 。
3. 如果 $k$ 不在内部节点 $x$ 中，那么寻找应该含有 $k$ 的子树 $x.c_i$ 。如果 $x.c_i$ 只有 $t - 1$ 个键，那么执行下面两步之一。最后，递归地在 $x.c_i$ 中删除 $k$ 。
   1. 如果 $x.c_i$ 只有 $t-1$ 个键，但是有一个紧邻 $x.c_i$ 的兄弟节点含有至少 $t$ 个键，那么从将 $x$ 中的一个键移动到 $x.c_i$ 中，然后从兄弟节点中移动一个键到 $x$ 中。
   2. 如果 $x.c_i$ 和 $x.c_i$ 的兄弟都只有 $t-1$ 个键，那么将 $x.c_i$ 和其中一个兄弟合并。
