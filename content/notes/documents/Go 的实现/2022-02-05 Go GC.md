> 本文是 <https://go.dev/blog/go15gc> 的部分翻译。

Go 正在构建的 GC （ Garbage Collector ，垃圾回收器）不仅是为 2015 年，更是为 2025 年以后。硬件能力的提升将会带来越来越大的吞吐量， Go 的 GC 为低延迟而设计，并且只能通过一个调节器来调节。

Go 的新垃圾回收器是一个*并发，三色，标记-清除*回收器，最早由 [Dijkstra 在 1978 年](http://dl.acm.org/citation.cfm?id=359655) 提出。这是与当今大多数“企业级”垃圾回收器的刻意区别，我们认为这很适合现代硬件的性质和现代软件的延迟要求。

在三色回收器中，所有对象都是白色、灰色或黑色的，我们把堆看作一个对象构成的图。在 GC 循环的开始，所有对象都是白色的。 GC 访问所有根节点，也就是所有应用程序可以直接访问的对象，例如全局变量和栈中的变量等。同时， GC 会把这些根节点标记为灰色。接下来 GC 选择一个灰色对象，将其变成黑色，扫描它是否含有指向其他对象的指针。当扫描到了一个指向白色对象的指针时， GC 将会把这个白色对象变成灰色。持续这个过程，直到没有灰色的对象。此时，白色的对象就是不可达的，可以背回收利用。

这一切都是与应用程序并发的，突变器（ mutator ，指的是与 GC 并发的任务）会在回收器运行时修改指针的值。因此，突变器必须保证不会将黑色对象中的指针指向白色对象，以免垃圾回收器失去对堆中已经访问过的对象的追踪。实现这种保证的方法就是**写屏障**（ write barrier ），突变器修改堆中指针时运行的一个小函数。 Go 的写屏障会把当前是白色的可达对象变成灰色，保证垃圾回收器最终会扫描其中的指针。

决定什么时候完成寻找所有灰色对象是很微妙的，如果我们想要避免阻塞突变器，可能会十分昂贵和复杂。为了让事情保持简单， Go 1.5 尽可能多地并发工作，然后短暂地 stop-the-world ，检查灰色对象的所有潜在来源。寻找 stop-the-world 所需的时间和 GC 的总工作量之间的最佳平衡点是 Go 1.6 的主要工作。

当然，困难之处在细节里。什么时候开始一个 GC 循环？根据什么指标来做出这个决定？ GC 应该怎样与 Go 调度器交互？怎样暂停突变器以便扫描其栈？如何表示白色、灰色和黑色以便我们快速找到灰色对象？如何找到根节点？如何找到对象中指针的位置？如何减少内存碎片？怎样处理缓存性能问题？堆应该多大？……这些问题中，有的与分配相关，有的与寻找可达对象相关，有的与调度相关，大多数与性能相关。

从高层来说，解决这些性能问题的方法之一是添加 GC 调节器，为每个性能问题添加一个调节器。编程者可以调节这些调节器，寻找最适合他们应用的设置。缺点是，每年添加一两个新调节器，最终会变成炼丹修仙的玄学。 Go 不想要走这条路。相反地， Go 只提供了一个调节器， GOGC 。它的值控制堆的总大小相对于可达对象的大学。默认值 100 表示堆的总大小比最后一次回收后可达对象的大小大 $100\%$ 。如果你想要 GC 的总时间变少，那么增加 GOGC 。如果你想要以提升 GC 时间为代价来减少内存使用，那么减少 GOGC 。

随着下一代硬件的 RAM 翻倍，简单地将 GOGC 翻倍即可减半 GC 循环数。另一方面，由于 GOGC 基于可达对象的大小，可达对象加倍造成的负载加倍并不需要调整 GOGC 。
